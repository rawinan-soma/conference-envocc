## Deferred from: code review of 4-5-booking-confirmation-link-token-qr (2026-06-15)

- **4.5-E2E-004 does not exercise the cross-organizer 403 path it names.** The skipped red-phase stub (`tests/e2e/bookings.spec.ts`) clears the session cookie and asserts only `status !== 200`, which tests the unauthenticated redirect, not the wrong-owner `assertOwner` 403. The stub's own comment acknowledges this ("A 403 test strictly requires a different authenticated user"). Both permit multiple NULLs in Postgres, so there is no runtime difference, but a future `drizzle-kit generate` may try to reconcile them (drop/recreate). The migration is hand-written by project convention (drizzle-kit generate is banned here), so the SQL is authoritative. Revisit if the schema is ever round-tripped through drizzle-kit.
- **Token-collision insert path (23505) is uncaught.** `createBooking` catches only `23P01` (exclusion_violation → ConflictError); a `registration_token` UNIQUE violation (`23505`) would surface as a raw 500. Probability for a 32-byte CSPRNG collision is negligible (~2^-128), so this is accepted. Revisit only if token entropy is ever reduced.
