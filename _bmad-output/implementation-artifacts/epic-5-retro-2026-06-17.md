## Challenges and Growth Areas

### 1. drizzle-kit generate full-schema dump (Story 5.2)

The drizzle-kit `generate` command produced a full-schema dump instead of an incremental migration (no intermediate snapshots 0001–0009 in meta/). The generated file had to be replaced with a hand-crafted incremental migration matching the spec pattern, and the journal tag updated manually. This pattern was already known from prior stories but still required manual intervention.

**Impact:** Extra time debugging migration generation vs. expectation.

**Root cause:** drizzle-kit's incremental snapshot mechanism requires continuous snapshot history. When snapshots are missing from meta/, it regenerates from the full current schema state.

### 2. Spec contradiction between security notes and service spec in Story 5.2

The story security notes said `cateringEnabled` should be re-validated against the DB record inside `createRegistration`, but the spec's own Service Spec code block did not include this re-validation — only `registrationEnabled` was re-read. The developer followed the code block (correct behavior), and the auditor rated the gap as defence-in-depth only (not an AC failure), but it introduced a deferred item that should be addressed: `cateringEnabled` DB precedence is not enforced at the service layer.

**Action item:** Tighten `createRegistration` to re-validate `mealType` against the DB `cateringEnabled` value in a future story.

### 3. Token-in-URL tradeoff for cancel links (5.3/5.4)

The cancel token is passed as a GET query parameter (`?token=<plaintext>`), which lands in access logs and browser history. This was an accepted by-design tradeoff (Dev Notes 5.4 line 110) — the URL contract was frozen by Story 5.3's PR #130. The mitigation (POST action moves token to form body, no cross-origin Referer leak for `/logo.svg`) was noted and the review dismissed it as intentional.

**Status:** By design, documented in deferred-work.md.

### 4. Close-registration handler status guard gap (Story 5.6)

The `close-registration` pg-boss handler fetches only `id`, `registrationEnabled`, and `registrationClosesAt` — not `booking.status`. If a booking is cancelled while `registrationEnabled=true` and a future close date is set, the handler will later fire and write `registrationEnabled=false` to an already-cancelled booking, creating a misleading audit trail. The consequence is benign (terminal state is still correct), but the audit noise is real.

**Root cause:** The handler spec did not include a `status` guard.

**Action item (deferred):** Either (a) extend the handler SELECT to include `status` and add `if (booking.status !== 'active') return;` as the first guard, or (b) have `cancelBooking` also set `registrationEnabled=false, registrationClosesAt=null`. Option (b) is the root-cause fix but touches Story 4.7 code. Product decision required.

### 5. Post-commit enqueueJob failure has no compensation (Story 5.6)

`enqueueJob` for the close-registration job is called after the DB transaction commits (correctly — pg-boss uses its own connection). But if `enqueueJob` throws (pg-boss down, queue not created), the booking is persisted without a scheduled auto-close. This is a genuine architectural hole with no compensation path.

**Action item (deferred):** Consider a transactional outbox or retry/reconcile path. Surface for Epic 6 or a dedicated hardening story.

### 6. Resend does not gate on registration-open/closed state (Story 5.5)

The resend action deliberately does not check whether registration is open or closed — it is a recovery mechanism for existing registrants and should work even after registration closes. This is a deliberate asymmetry but was documented as worth revisiting if product decides resend must also be gated.

**Status:** Deliberate design decision. Deferred pending product direction.

## Key Insights

1. **The CSPRNG token + sha256 hash pattern is a reliable building block.** Stories 5.2, 5.4, and 5.5 all used it with no issues. The single-use enforcement via `cancel_token_hash = NULL` worked cleanly with `FOR UPDATE` row locks.

2. **Parallel story execution after a foundation story is an effective pattern.** The 5.1 → 5.2 → (5.3/5.6/5.7/5.8 parallel) → (5.4/5.5) dependency structure was correctly identified and executed, completing all 8 stories in 3 days.

3. **Spec code blocks should be the authority when prose and code conflict.** The 5.2 cateringEnabled discrepancy between security notes (prose) and Service Spec (code block) was correctly resolved by following the code block. Future stories should flag similar contradictions during planning, not during review.

4. **pg-boss worker import boundary (R-011) requires vigilance.** The worker is a standalone Bun process; `$lib` and `$app/*` aliases are not available. The rule is documented but easy to forget. The `5.6-INT-005` lint test enforces it in CI, which is the correct mitigation.

5. **Zero Svelte 4 syntax debt accumulated in Epic 5.** All new Svelte components used Svelte 5 runes (`$props()`, `$derived`, `$state`) throughout. No `$:` reactive declarations or `export let` syntax.

6. **The `deferred-work.md` file remains the right home for architectural follow-ups.** 6 Epic 5 items were added. The file now carries 13 items from across the project — a natural checkpoint for future hardening.
