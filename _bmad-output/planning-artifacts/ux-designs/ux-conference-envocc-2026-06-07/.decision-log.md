| UXD-026 | 2026-06-09 | Registration (internal) | **Internal registration is a distinct experience from external — not a variant of the same page.** An internal user can register to **attend** an event **in-app** (not via the public token link). Their identity (name, organization, email) is **auto-filled read-only from their app profile**; the **only input is meal type**, shown only when catering is enabled — often a single confirm tap. **Entry point: a new Event Detail view** opened by clicking a booking chip on the Room Calendar → **"Register to attend"** action. The public token registration page stays **anonymous/unauthenticated (FR-045 intact)** for external guests. **Not a new role** — every internal user can act as **organizer and/or attendee** (same account, two capacities). Internal confirmation is an **in-app toast/inline** state, not the branded public confirmation screen; self-cancel from the same Event Detail. | Rawinan: "When internal staff register to the event it should not fill in any form but the meal if provided, because the data comes from app database." Chose **in-app "Register to attend."** Clarified: "not a new role, everyone could be either organizer or attendee." |

## Open Questions

| ID | Question | Status |
|----|----------|--------|
| OQ-1 | **FR-094 visibility relaxation (PRD dependency for UXD-026).** To register in-app, an internal user must be able to **view another organizer's event detail** (name, date, time, room, agenda, contact). FR-094 currently limits organizers to their **own** bookings. Need a PRD update: internal users may **view any event for the purpose of attending/registering** (read-only); **manage/edit stays own-only**, admin read-all unchanged. | **Resolved 2026-06-09** — PRD relaxed FR-094 (PRD DEC-025): internal users may view another organizer's event read-only (event details only, not registrant data) to register; manage/edit stays own-only; admin read-all unchanged. |
| OQ-2 | **Profile data for prefill (PRD/architecture dependency for UXD-026).** Building an internal registrant record needs **email + organization**. FR-095 currently guarantees only **name + phone**. The internal profile must carry email + organization. Ties to existing **OQ-3** (profile source — self-service vs. org IdP — deferred to architecture). | **Resolved 2026-06-09** — PRD expanded FR-095 (PRD DEC-027) to {title, name, phone, email, organization}; internal registrant records carry a title. The remaining "how populated (IdP vs self-service)" question stays as PRD OQ-3, deferred to architecture. |
